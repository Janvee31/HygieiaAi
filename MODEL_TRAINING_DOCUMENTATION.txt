Important note about GitHub
---------------------------
Large datasets and trained model artifacts should not be pushed directly to a normal GitHub repository.

GitHub warns when a file is larger than 50 MiB and blocks regular Git files larger than 100 MiB. This project includes large datasets, image folders, notebooks, and PyTorch model files. These are better shared separately through a pendrive, external drive, Google Drive, OneDrive, Hugging Face, Kaggle, or Git LFS.

Recommended sharing approach:
1. Push only source code, SQL files, and documentation to GitHub.
2. Keep backend\data, backend\model, backend\models, backend\saved_models, and large notebooks outside normal Git.
3. Share the large folders separately with the friend who needs to run the full project.


1. General Disease Prediction
-----------------------------
Training script:
backend\training\train_disease_model.py

Dataset used:
backend\data\clean_dataset.tsv

Target:
Disease label, encoded using sklearn LabelEncoder.

Model actually trained:
XGBoost Classifier

Important code choices:
- xgboost.XGBClassifier
- objective = multi:softprob
- GridSearchCV used for hyperparameter tuning
- Parameters tuned include max_depth, learning_rate, n_estimators, subsample, and colsample_bytree
- LabelEncoder saved as disease_label_encoder.pkl
- Model saved as disease_prediction_model.json

Why XGBoost was used:
General disease prediction is a multi-class tabular classification problem based on many symptom indicators. XGBoost is a strong choice for this because it handles structured tabular data very well, captures non-linear relationships between symptoms, supports multi-class probability output, and usually performs better than a single decision tree or simple linear model on mixed medical-style feature sets.

Why this model is suitable here:
- Good for tabular symptom features.
- Handles many input columns.
- Gives probability-style output for multiple disease classes.
- Provides feature importance, useful for explaining which symptoms influenced the prediction.
- Works well with moderate-sized medical datasets without needing a deep neural network.


2. Diabetes Prediction
----------------------
Training script:
backend\training\train_diabetes_model.py

Dataset used:
backend\data\diabetes.csv

Target:
Outcome

Models actually trained:
- Logistic Regression
- Random Forest Classifier
- Gradient Boosting Classifier
- Support Vector Machine
- Soft Voting Ensemble using the above models

Important code choices:
- Missing zero values in Glucose, BloodPressure, SkinThickness, Insulin, and BMI are replaced with median values.
- StandardScaler is used in model pipelines.
- GridSearchCV is used for model tuning.
- VotingClassifier uses soft voting.
- Best model is selected from trained candidates and saved as the default diabetes model.

Why these models were used:
Diabetes prediction is a small-to-medium tabular binary classification task. The script trains multiple model families because each one captures a different type of pattern:

Logistic Regression:
- Good baseline model.
- Easy to interpret.
- Useful when medical indicators have mostly linear relationships with risk.

Random Forest:
- Handles non-linear feature interactions.
- Robust to noisy input.
- Gives feature importance.

Gradient Boosting:
- Often performs strongly on structured tabular datasets.
- Learns difficult decision boundaries by combining weak learners.

SVM:
- Useful for smaller datasets.
- Can model non-linear separation with kernels.

Soft Voting Ensemble:
- Combines the strengths of all four models.
- Reduces dependence on a single model's weakness.
- Can improve stability when individual model performance is close.

Why the final choice is made this way:
The script compares model performance and saves the best-performing model. This is practical because diabetes risk features can behave differently depending on preprocessing and dataset distribution.

backend\training\train_kidney_disease_model.py

Dataset used:
backend\data\chronic_kidney_dataset.csv

Models actually trained:

Preprocessing actually used:
- KNNImputer for numeric values
- SimpleImputer for categorical values

- StratifiedKFold cross-validation
- GridSearchCV using ROC AUC
- Best model selected by AUC
- Best model saved as kidney_disease_model.pkl

Why these models were used:
Kidney disease datasets usually contain a mix of numeric lab values and categorical clinical values. The script uses a full preprocessing pipeline because missing values and mixed column types are expected.

Random Forest:
- Handles mixed patterns and non-linear relationships.
- Good for clinical tabular data.

Logistic Regression:
- Provides an interpretable baseline.
- Useful for checking whether simple weighted clinical indicators are enough.

Gradient Boosting:
- Strong performance on tabular medical data.
- Can learn complex risk patterns.

SVM:
- Useful for binary classification after scaling.
- Provides a different decision boundary compared with tree models.

Voting Ensemble:
- Combines different classifiers to improve robustness.
- Helps when no single model is consistently dominant.

Why this approach is suitable:
Kidney disease prediction needs robust missing-value handling and comparison across several model families. Selecting the final model by AUC is appropriate because disease datasets can be imbalanced and accuracy alone may be misleading.

7. Stroke Prediction
--------------------
Training script:
backend\training\train_stroke_model.py

Dataset used:
Synthetic data generated in the script using sklearn.datasets.make_classification.

Target:
Binary stroke risk label.

Model actually trained:
Random Forest Classifier inside a Pipeline

Important code choices:
- make_classification creates 1000 samples.
- 10 features are generated and then scaled into realistic-looking values.
- StandardScaler is used.
- RandomForestClassifier uses n_estimators=50, max_depth=8, class_weight='balanced'.
- Model saved as backend\saved_models\stroke_model.pkl.

Why Random Forest was used:
Stroke risk depends on several interacting risk factors, such as age, hypertension, heart disease, glucose, BMI, and smoking status. Random Forest can model these interactions and is less sensitive to individual noisy features.

Why balanced class weight was used:
Stroke is represented as a minority class in the generated data. The class_weight='balanced' setting helps the model pay more attention to positive stroke-risk samples instead of mostly learning the majority no-stroke class.

Important limitation:
This script trains on synthetic data, not a real clinical stroke dataset. It is suitable for demonstration and application flow testing, but not for real medical use.

9. Breast Cancer Image Detection
--------------------------------
Training scripts:
backend\training\train_breast_cancer_image_model.py
backend\training\train_breast_cancer_simplified.py

Dataset used:
backend\data\Breast Cancer

Targets:
Binary classification:
- BENIGN
- MALIGNANT

Models actually used:
- EfficientNet B0 in train_breast_cancer_image_model.py for normal mode
- MobileNetV2 in train_breast_cancer_image_model.py for fast mode
- ResNet18 in train_breast_cancer_simplified.py

Important code choices:
- PyTorch and torchvision are used.
- Transfer learning is used with pretrained CNN backbones.
- Final classifier layer is replaced for binary classification.
- CrossEntropyLoss is used.
- Adam optimizer is used.
- ReduceLROnPlateau scheduler is used.
- Data augmentation includes resize, random horizontal flip, random rotation, color jitter, normalization.
- Mixed precision training is supported.
- Models are saved in .pth and full .pth formats for inference.

Why EfficientNet B0 was used:
EfficientNet B0 gives strong image classification performance with fewer parameters than many older CNNs. It is a good choice for mammography-style classification where image features can be subtle and transfer learning is useful.

Why MobileNetV2 was used in fast mode:
MobileNetV2 is lightweight and trains faster. It is useful for hackathon/demo mode or machines with limited GPU resources.

Why ResNet18 was used in the simplified script:
ResNet18 is a reliable transfer learning baseline. It trains quickly, is easier to debug, and performs well enough for a demo medical image classifier.

Why transfer learning was used:
Medical image datasets are often not large enough to train a CNN from scratch. Pretrained ImageNet backbones provide reusable low-level visual features such as edges, textures, and shapes, which can be fine-tuned for medical classification.

13. Model Selection Strategy Used Across the Project
----------------------------------------------------
The project uses two main strategies:

Tabular disease models:
- Train several classical machine learning models.
- Use preprocessing pipelines.
- Use cross-validation and GridSearchCV.
- Compare using accuracy or ROC AUC.
- Save the best model as the default model for prediction.

Image disease models:
- Use transfer learning with pretrained CNNs.
- Replace the final classification layer for the required disease classes.
- Use augmentation to improve generalization.
- Use Adam optimizer and learning-rate scheduling.
- Save models in PyTorch formats for inference.


14. Why Not Use One Huge Model for Everything?
----------------------------------------------
A single universal model was not used because each prediction task has different data:

- Diabetes, heart, liver, kidney, lung, stroke, and thyroid use tabular numeric/categorical data.
- Breast cancer, skin cancer, and brain tumor use images.
- General disease prediction uses symptom vectors and has many output classes.

Different data types require different model families. Tree-based models and linear/SVM models are better for structured tabular data, while CNN transfer learning models are better for image data.
